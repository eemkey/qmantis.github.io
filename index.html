Case Study

1. Introduction

QMantis is a GraphQL server with an autoinstrumentation solution for GraphQL APIs using Express. It's an easy and quick way to get your Express-based GraphQL API backend running with an observability feature, which generates and exports metrics and traces data and creates a Grafana dashboard for you to analyze.

5.3 Architecture

This diagram gives an overview of the QMantis architecture. There are two main overarching and interconnected components: qmantis-express and qmantis-compose. On the left, qmantis-express is a npm package with a configured express-graphql server that is responsible for generating the metrics and traces. On the right, qmantis-compose lets the user deploy the QMantis architecture on their own server using Docker in order to process, export, store, and visualize incoming metrics and traces.

Now, to provide a better understanding, let's break up this architecture into key phases.

The QMantis architecture is split up into four major conceptual phases:
Phase 1: Generate Traces and Metrics with a GraphQL server and OpenTelemetry SDKs
Phase 2: Export Data for Processing and Storage
Phase 3: Store Data with Promscale / TimescaleDB
Phase 4: Visualize / Analyze Data with Grafana

6. Design Decisions

Let's now go over some of the important design decisions we made while developing QMantis.

6.1 Grafana vs Custom UI

When it came to visualizing data, our team considered two options - building a custom user interface or using Grafana. A custom UI would have provided two main benefits: we could have branded the UI to add a QMantis-specific design, and we could have displayed data more intuitively - especially for a first time user. On another hand, we would have to maintain a whole extra code base and any updates to the UI would demand a lot of time.

Grafana, an industry standard observability tool that many developers are already familiar with, is extremely customizable. Once a user has QMantis running, they can change the dashboard and query settings to best fit their needs. To give an example, projects at different points in the development cycle may want to query data by different lengths of time, and Grafana makes that very easy. Moreover, if our team decide to update the dashboard or panels in the future, reconfiguration would be relatively simple.

Ultimately, the pros of building a custom UI did not outweigh the pros of using Grafana, so we chose to go with Grafana.

6.2 GraphQL server choice

Early on in the process of building QMantis, our team considered building our own GraphQL server. This would have given us control over the implementation of the server, and made it simple to generate and collect data exactly how we wanted. However, if a developer had an already existing GraphQL API, they would have had to change their code to work with our less popular server in order to gain observability. Taking this into consideration, we decided it would be more useful to build a tool that could work with an already widely used GraphQL server.

Another option was to use Apollo Server, a popular GraphQL server. Note that this is different than the fully managed Apollo Studio observability service. Our team considered using Apollo Server to give users the option to use a GraphQL server that was framework-agnostic for Node.js. However, this option had a major downside -- the way Apollo Server is implemented makes it very difficult to extract meaningful trace information, even when creating custom plugins to extract this information. Moreover, Apollo Studio, from the same company, currently offers observability for a limited amount of metrics for free, and more detailed observability such as traces with a paid subscription when using their server.

In the end, we landed on Express GraphQL. It's the second most popular JavaScript server, used by millions of developers around the world and, unlike Apollo, Express GraphQL does not currently have any out of the box observability options - paid or unpaid. With this in mind we chose to build our first iteration of QMantis to be designed to work with Express Graphql -- giving it ability to provide observability for a popular GraphQL server with minimal changes to a codebase.

7. Future Work

7.1 Generating metrics data from express-graphql

Choosing to use express-graphql as our GraphQL server came with significant challenges. Because we didn't have control over the implementation of the server, we needed to be creative in order to collect the data we needed from the request and response objects. That information was essential to generate the accurate and GraphQL-specific telemetry data, which is a key QMantis feature.

Counting errors and collecting latency data when status codes was 4xx or 5xx were the two most difficult pieces of data to obtain. To get the accurate information, we needed to access the response object before it was sent back to the client. To do that, we were relying on a function from the express-graphql server to access the response object. However, when the status code was 4xx or 5xx, the express-graphql server doesn't execute that function, so we were unable to check for errors or get the latency information we needed.

We considered modifying our architecture and even prototyped a GraphQL server. This approach worked and we were able to obtain the data we needed, but it came with the limitations mentioned previously. Ultimately, we found a solution to our challenges: we used an existing npm package called response-time to obtain the latency data and relied on trace data to accurately count errors.

7.2 Connecting metrics with traces

Another challenged we face was to connect metrics with traces. Initially, we wanted to connect metrics with traces to make it easy for the developer to go from spotting a potential problem to get the detailed trace information. How would that work? When the developer spots something worth investigating, such as an outlier metric datapoint, an increase in the error rate, they would be able to click on the datapoint and it would take them to a list of traces that occurred within that time frame.

After a lot of research, we found out that the only way to have this feature in Grafana was to use Grafana Tempo, Grafana's own solution for traces backend. Currently, Grafana Tempo is only compatible with an object storage such as Google Cloud Storage, Amazon S3 or Azure Blob Storage. That meant we would need a different database to store our traces data, adding extra complexity to our architecture. Another option was to build our own UI, which wasn't ideal, as discussed previously.

This left us with the challenge of how to make it easy to go from metrics to traces in our dashboard. To solve it, we came up with two traces panels, one that shows traces by latency, making it easy to spot the outliers. The other one, only shows traces that contains errors. As both metrics and traces have a time-stamp, the user can quickly filter the traces panels to look for the needed information. Therefore, our solution makes it easy to investigate problems shown on metrics without the added complexity of a second database.

It is also worth noting that, as of early 2022, the leading paid observability solutions for GraphQL do not currently support linking metrics and traces together.

9. Future Work

9.1 Expanding Metrics

Our team believes adding certain datasets to the dashboard could improve the usefulness of QMantis. Namely, we would like a developer to be able to see how often certain operations are being requested and what the average latency of a specific operation is. These features would allow a developer to get a quick read on the efficiency of individual operations and prioritize potential problems based on how often an operation is being requested.
